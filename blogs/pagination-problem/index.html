Paging 2 Triệu Dòng Dữ Liệu — Tại sao OFFSET chết và giải pháp thay thế

Câu hỏi phỏng vấn Senior chắc chắn gặp. Câu trả lời không phải là "thêm RAM" hay "tăng instance" — mà là hiểu cách database thực sự hoạt động bên trong.

Thong Nguyen 24 tháng 3, 2026

Khi dữ liệu chỉ có vài nghìn dòng, mọi cách paging đều chạy nhanh. Vấn đề chỉ lộ ra khi bảng có hàng triệu dòng và user cố nhảy đến trang cuối. Lúc đó, cách làm thông thường sẽ khiến query chạy hàng chục giây — hoặc timeout hoàn toàn.

Vấn đề cốt lõi: Tại sao OFFSET lại "chết"?

Cách làm quen thuộc của hầu hết developer khi mới bắt đầu:

-- Lấy trang thứ 50,001 (mỗi trang 20 dòng)
SELECT * FROM users
ORDER BY created_at DESC
LIMIT 20 OFFSET 1000000;
❌ Sai lầm chết người: Database không có phép thuật để "nhảy bùp" tới dòng thứ 1 triệu. Nó phải scan toàn bộ 1 triệu dòng đầu tiên, đếm đủ, vứt đi — rồi mới lấy 20 dòng tiếp theo. Trang càng về sau, query càng chậm theo cấp số nhân. OFFSET 1,000,000 ở một bảng không được tối ưu có thể mất 10–30 giây.
Scan 1,000,000 dòng
→
Đếm đủ 1M
→
Vứt hết 1M dòng
→
Trả về 20 dòng

Đây gọi là bài toán Deep Pagination. Dữ liệu càng lớn, trang càng sâu, hệ thống càng nguy hiểm.

⚠️ Bài toán kép — COUNT(*) cũng chậm: Để hiển thị "Trang 3 / 50,000", bạn cần biết tổng số trang, tức là phải chạy SELECT COUNT(*) FROM users WHERE .... Trên bảng 2 triệu dòng với điều kiện lọc phức tạp, câu lệnh COUNT này cũng có thể mất vài giây — và thường bị developer bỏ qua khi tối ưu.

Giải pháp 1 — Keyset Pagination (Cursor-based)

Dành cho: Infinite Scroll, Load More, Next/Prev — không cần nhảy trang tự do

Thay vì nói "bỏ qua N dòng đầu", ta nói "lấy các dòng sau cái id/timestamp cuối cùng tôi đã thấy". Database lúc này chỉ cần dùng Index để tìm đến đúng điểm đó và nhặt ra 20 dòng — không cần scan gì thêm.

-- Trang đầu tiên
SELECT * FROM users
ORDER BY id DESC
LIMIT 20;

-- Trang tiếp theo (last_seen_id = ID của dòng cuối trang trước)
SELECT * FROM users
WHERE id < :last_seen_id
ORDER BY id DESC
LIMIT 20;
✅ Tại sao siêu nhanh?
Cột id có B-Tree Index. Database dùng Index để tìm trực tiếp đến last_seen_id với độ phức tạp O(log n), rồi duyệt tuần tự lấy 20 dòng tiếp theo. Tốc độ trang 1 và trang 100,000 là gần như bằng nhau.

Bổ sung Sort key phải unique — tránh bẫy timestamp trùng

Nếu bạn sort theo created_at thay vì id, hãy cẩn thận: nhiều dòng có thể có cùng timestamp (insert trong cùng 1 giây). Cursor created_at < last_seen_at sẽ bỏ sót dữ liệu.

-- ❌ Nguy hiểm: created_at có thể trùng
WHERE created_at < :last_seen_at

-- ✅ An toàn: dùng composite cursor (created_at + id)
WHERE (created_at, id) < (:last_seen_at, :last_seen_id)
ORDER BY created_at DESC, id DESC
LIMIT 20;
Index cần khớp với sort key: Để query trên chạy nhanh, cần composite index: CREATE INDEX idx_users_cursor ON users (created_at DESC, id DESC);. Index này được gọi là Covering Index — toàn bộ thông tin cần cho việc lọc đều nằm trong Index, không cần đụng vào bảng thật.

Bổ sung Phân trang 2 chiều (Previous page)

Với Keyset Pagination, "trang trước" không đơn giản như trang sau. Cách phổ biến nhất là lưu 2 cursor trong response: một cho "next" và một cho "prev".

-- Response trả về:
{
  "data": [...],
  "cursor": {
    "next": "eyJpZCI6MTAwfQ==",   // base64({"id": 100})
    "prev": "eyJpZCI6MTIwfQ=="    // base64({"id": 120})
  },
  "has_next": true,
  "has_prev": true
}

-- Query "trang trước": đảo điều kiện và order, lấy rồi reverse kết quả
SELECT * FROM users
WHERE id > :prev_cursor_id
ORDER BY id ASC
LIMIT 20;

Giải pháp 2 — Deferred Join (Late Row Lookup)

Dành cho: UI có nút số trang (1, 2, 3 ... 999) — bắt buộc phải dùng OFFSET

Khi business yêu cầu "phải có nút nhảy trang", bạn buộc phải dùng OFFSET. Nhưng thay vì để database scan hàng triệu dòng dữ liệu đầy đủ, ta tách thành 2 bước: scan nhanh trên Index để lấy ID, rồi mới lấy dữ liệu đầy đủ từ bảng thật cho đúng 20 dòng đó.

-- ❌ Cách thông thường: scan 1M dòng đầy đủ (nặng)
SELECT * FROM users
ORDER BY created_at DESC
LIMIT 20 OFFSET 1000000;

-- ✅ Deferred Join: scan 1M trên Index (nhẹ), chỉ đọc 20 dòng từ bảng thật
SELECT u.* FROM users u
INNER JOIN (
  SELECT id FROM users
  ORDER BY created_at DESC
  LIMIT 20 OFFSET 1000000
) AS q ON u.id = q.id;
Tại sao nhanh hơn? Subquery bên trong chỉ SELECT mỗi cột id. B-Tree Index của created_at chứa kèm id, nên toàn bộ subquery chạy 100% trong Index (in-memory), không cần đọc disk. Sau đó mới dùng 20 ID đó để JOIN lấy dữ liệu đầy đủ. Thay vì vứt 1 triệu dòng đầy đủ, chỉ vứt 1 triệu cái ID bé tí.

Bổ sung Giải quyết COUNT(*) chậm

Bài toán kép: hiển thị tổng số trang nhưng COUNT(*) trên 2 triệu dòng cũng chậm. Ba hướng giải quyết theo thứ tự ưu tiên:

  1. 1 Cache tổng số dòng: Đừng COUNT mỗi request. Lưu tổng số dòng vào Redis, cập nhật khi insert/delete. Chấp nhận con số hơi lệch vài giây (eventual consistency) — user không cần biết chính xác "1,234,567 kết quả" hay "1,234,580 kết quả".
  2. 2 Approximate count — Đếm xấp xỉ: Đây là kỹ thuật dùng số ước tính có sẵn trong đầu não của database — thay vì đếm lại từ đầu. Mỗi database đều định kỳ tự động thống kê bảng (được gọi là table statistics) để tối ưu query planner. Trong quá trình đó, nó ước tính toàn bộ số dòng và lưu vào catalog nội bộ. Ta lấy con số đó ra mà không cần đếm gì thêm — gần như tức thì, bất kể bảng có bao nhiêu dòng. Đổi lại, con số có thể lệch vài % so với thực tế — hoàn toàn chấp nhận được khi hiển thị “Khoảng 1.2 triệu kết quả”. 
-- PostgreSQL: lấy approximate count từ system catalog, gần như instant
SELECT reltuples::bigint AS approx_count
FROM pg_class
WHERE relname = 'users';

-- MySQL / MariaDB: tương tự, lấy từ information_schema
SELECT TABLE_ROWS AS approx_count
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'your_db'
  AND TABLE_NAME = 'users';

-- MongoDB: tương tự, estimatedDocumentCount() thay vì countDocuments()
-- countDocuments(): đếm chính xác, chậm trên collection lớn
-- estimatedDocumentCount(): đọc metadata, gần như instant
db.users.estimatedDocumentCount()
Cơ chế hoạt động: PostgreSQL chạy tác vụ ANALYZE định kỳ (hoặc tự động qua autovacuum) để cập nhật table statistics. MySQL tương tự với ANALYZE TABLE. Con số reltuples / TABLE_ROWS sau đó được cập nhật — thường chỉ lệch từ vài % đến dưới 10% so với thực tế. Trên bảng có write traffic cao, nên trigger ANALYZE sau khi insert batch lớn để số liệu luôn gần đúng.
  1. 3 Giới hạn số trang hiển thị: Google chỉ hiển thị tối đa 10 trang kết quả (khoảng 100 kết quả). Thực tế, hơn 99% user không bao giờ vượt quá trang 5. Giới hạn MAX_PAGE = 100 loại bỏ hoàn toàn bài toán deep paging mà không ai phàn nàn.

Giải pháp 3 — Elasticsearch cho Search & Paging phức tạp

Khi dữ liệu lên tới hàng chục triệu dòng và user cần full-text search (tìm theo từ khoá, fuzzy matching, relevance ranking) — SQL không còn là công cụ phù hợp nữa. Đây là lúc cần Elasticsearch.

Elasticsearch cũng có vấn đề Deep Paging: Cơ chế from + size của ES (tương đương OFFSET) cũng giới hạn ở 10,000 kết quả mặc định (max_result_window). ES giải quyết bằng 2 API: search_after (giống Keyset Pagination) cho real-time search, và Scroll API cho batch processing.
  1. 1 Đồng bộ dữ liệu: Dùng CDC (Change Data Capture) — các tool như Debezium đọc binlog của MySQL/PostgreSQL và stream dữ liệu thay đổi lên Kafka → Elasticsearch theo thời gian thực. Không cần viết code đồng bộ thủ công.
  2. 2 search_after cho real-time paging: Giống Keyset Pagination, dùng sort values của dòng cuối cùng làm cursor cho request tiếp theo. Tốc độ ổn định dù data lên hàng tỷ document.
  3. 3 Scroll API cho export/batch: Khi cần xử lý toàn bộ dữ liệu (export CSV, reindex), dùng Scroll API — ES giữ một snapshot của kết quả search và cho phép duyệt qua từng batch mà không bị affected bởi dữ liệu đang thay đổi.
✅ Nguyên tắc: Dùng SQL cho dữ liệu có cấu trúc, query đơn giản, volume vừa phải. Dùng Elasticsearch khi cần full-text search, faceted filtering (lọc theo nhiều chiều như Shopee/Tiki), hoặc volume từ chục triệu dòng trở lên.

Bổ sung — Paging cho Background Jobs

Một bài toán hay bị bỏ qua: khi cần xử lý toàn bộ 2 triệu dòng trong một background job (ví dụ: gửi email hàng loạt, migrate dữ liệu), đừng bao giờ dùng pagination thông thường.

-- ❌ Sai: dùng OFFSET trong job, càng về sau càng chậm
LOOP:
  SELECT * FROM users LIMIT 1000 OFFSET :current_offset
  current_offset += 1000

-- ✅ Đúng: cursor-based iteration, tốc độ ổn định từ đầu đến cuối
last_id = 0
LOOP:
  batch = SELECT * FROM users
          WHERE id > last_id
          ORDER BY id ASC
          LIMIT 1000
  IF batch.empty: BREAK
  process(batch)
  last_id = batch.last.id
Lợi ích kép: Cursor iteration không chỉ nhanh hơn mà còn resumable — lưu last_id vào Redis trước khi xử lý mỗi batch. Job bị crash giữa chừng? Khởi động lại từ đúng điểm đó mà không mất dữ liệu, không process trùng.

Tóm tắt — Quyết định nào cho tình huống nào?

Tình huống Giải pháp
Feed, Infinite Scroll, Load More Keyset Pagination với composite cursor

Không có giải pháp nào là "tốt nhất" cho mọi trường hợp. Senior Engineer hiểu rõ trade-off của từng lựa chọn và chọn đúng công cụ cho đúng bài toán — thay vì cứ OFFSET vì đó là thứ họ học đầu tiên.
