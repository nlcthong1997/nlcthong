Upload 10GB & Stream cho 1000 Users — Hệ thống không sập như thế nào?

Bí quyết cốt lõi không phải là làm cho Backend "chịu đựng" được file khổng lồ, mà là đừng để nó phải chạm vào.

T
Thong Nguyen 24 tháng 3, 2026

Đây là bài toán kinh điển (System Design) mà YouTube, Google Drive, Netflix đã giải quyết. Tư duy ở cấp độ Senior không phải là "nhồi nhét" cho Backend chịu được file 10GB — mà là thiết kế để Backend không cần chạm vào nó.

Upload file 10GB làm sao để Backend không sập?

❌ Sai lầm thường gặp: Client đính kèm file 10GB vào multipart/form-data và gửi thẳng lên Backend.
Kết quả: Backend nạp toàn bộ file vào RAM → Out of Memory, hoặc request timeout giữa chừng do mạng chậm, đồng thời block toàn bộ thread xử lý khác.

Giải pháp 1 Presigned URL — Upload thẳng lên Cloud

Cách tốt nhất để Backend không sập là… cho Client tự upload thẳng lên nơi lưu trữ như AWS S3 hoặc Google Cloud Storage, bỏ qua Backend hoàn toàn.

① Client xin phép
→
Backend kiểm tra quyền
→
Tạo Presigned URL
→
② Client nhận link
③ Upload thẳng lên S3
→
S3 gửi Webhook
→
④ Backend cập nhật DB
  1. 1 Xin phép (Request): Client gọi API lên Backend báo "Tôi muốn upload video 10GB".
  2. 2 Cấp vé (Presigned URL): Backend kiểm tra quyền (đã đăng nhập? đủ dung lượng?). Nếu OK, dùng Secret Key gọi lên S3 tạo một Presigned URL — đường link upload tạm thời, hiệu lực 15 phút — rồi trả về cho Client.
  3. 3 Upload trực tiếp: Client dùng link đó gửi thẳng file 10GB lên S3. Backend lúc này hoàn toàn rảnh rỗi phục vụ người khác.
  4. 4 Thông báo (Webhook): Khi S3 nhận đủ 10GB, nó bắn Webhook (hoặc Message Queue) báo Backend: "Client A đã upload xong file X". Lúc này Backend mới cập nhật Database.
✅ Lợi ích: Backend không hề nhìn thấy byte nào của file 10GB. Không OOM, không timeout, không block thread — hoàn toàn scale độc lập.
🔒 Presigned URL có an toàn không? Câu hỏi thường gặp: "Ai cũng có link thì upload bừa được không?" — Không. Link được ký bằng HMAC với Secret Key của AWS/GCS, có TTL giới hạn (thường 15 phút), chỉ có thể upload đúng vào một path cụ thể đã chỉ định, và không thể dùng để đọc hay xoá file khác. Hết hạn hoặc sai chữ ký → S3 từ chối ngay lập tức với lỗi 403 Forbidden.

Giải pháp 2 Chunked Upload — Băm nhỏ file

Nếu không dùng Cloud mà bắt buộc lưu file ở server nội bộ (On-premise), bạn phải dùng cơ chế băm nhỏ file ở phía Frontend.

  1. 1 Slicing: Dùng JavaScript File.slice() cắt file 10GB thành hàng nghìn mảnh nhỏ (chunk), mỗi mảnh ~5MB.
  2. 2 Gửi tuần tự / song song: Client gửi từng chunk 5MB lên Backend. Backend nhận mảnh nào, dùng Stream ghi nối tiếp (append) ngay xuống ổ cứng. RAM chỉ tốn tối đa 5MB mỗi request — không bao giờ sập.
  3. 3 Resumable: Mạng đứt tại chunk thứ 500? Client chỉ cần gửi tiếp từ chunk 501. Cực kỳ an toàn, không mất dữ liệu.
Tip: Các hệ thống lớn thường dùng giao thức mã nguồn mở TUS (tus.io) để chuẩn hoá Chunked Upload với đầy đủ tính năng resumable, checksum và metadata.

Bổ sung Validate file sau khi upload — Đừng bỏ qua!

Vì Backend không trực tiếp "chạm" vào file, câu hỏi đặt ra là: ai sẽ kiểm tra virus, malware hay nội dung độc hại? Đây là bước cực kỳ quan trọng mà nhiều hệ thống bỏ sót.

⚠️ Rủi ro bị bỏ qua: Nếu không validate, user có thể upload file .exe, script độc hại, hoặc nội dung vi phạm. Chỉ cần một file lọt qua là đủ gây hậu quả nghiêm trọng cho toàn hệ thống và người dùng khác.
  1. 1 Trạng thái PENDING: Ngay khi file upload xong, ghi vào DB với trạng thái PENDING — chưa cho phép truy cập công khai.
  2. 2 Trigger scan job: S3 Event Notification kích hoạt một Lambda / Worker chạy virus scan (ClamAV, AWS GuardDuty Malware Protection).
  3. 3 Cập nhật trạng thái: Scan sạch → DB chuyển thành READY, cho phép truy cập. Phát hiện mối đe doạ → chuyển sang QUARANTINED, xoá file và cảnh báo admin.

Bonus Upload xong → Video đi đâu? Luồng hoàn chỉnh

Sau khi S3 nhận xong file và Webhook bắn về BE, hệ thống cần thêm một bước nữa trước khi video sẵn sàng để xem — đây là cầu nối giữa hai vế của bài toán.

Tại sao cần Message Queue (SQS / RabbitMQ / Kafka)? Thay vì BE tự chạy FFmpeg và bị treo, nó chỉ đẩy một message nhỏ vào Queue rồi trả về ngay. Worker pool (scale độc lập) lần lượt xử lý các job transcoding — hệ thống không bao giờ bị nghẽn dù có hàng nghìn video upload cùng lúc.

1000 Users cùng xem video 10GB làm sao không sập?

❌ Sai lầm thường gặp: Lưu file video.mp4 10GB trong ổ cứng server, rồi trả nguyên đường link cho user. 1000 user tải cùng lúc = 10 Terabytes băng thông. Card mạng server bị cháy, video cứ xoay vòng (buffering) mãi không chạy.

Giải pháp 1 Video Transcoding & HLS / DASH

Bạn không bao giờ được phục vụ file gốc 10GB cho người xem. Ngay sau khi upload xong, file phải được ném vào một Background Job (dùng FFmpeg chẳng hạn) để xử lý.

  1. 1 Transcoding: Nén và render video gốc thành nhiều độ phân giải: 1080p, 720p, 480p, 360p.
  2. 2 Segmentation: Cắt mỗi version thành các đoạn cực nhỏ, dài 2–6 giây (.ts hoặc .m4s).
  3. 3 Manifest file: Tạo file .m3u8 liệt kê thứ tự các đoạn video nhỏ kia.
Adaptive Bitrate Streaming: Player của user tải file .m3u8 trước, rồi cứ kéo từng đoạn 2 giây về phát. Mạng khoẻ → tải đoạn 1080p. Mạng yếu → tự hạ xuống 480p. Server không cần "ném" cục 10GB đi nữa — mỗi request chỉ là vài MB.
📡 User biết khi nào video sẵn sàng? Transcoding một video 10GB có thể mất vài phút. Có hai cách thông báo phổ biến: Polling — client tự gọi API kiểm tra trạng thái mỗi 5 giây (đơn giản, dễ làm, nhưng tốn request); hoặc WebSocket / Server-Sent Events (SSE) — server chủ động đẩy thông báo "Video của bạn đã sẵn sàng!" ngay khi transcoding xong (real-time, trải nghiệm mượt hơn). YouTube sử dụng cách thứ hai.

Giải pháp 2 CDN — Sát thủ chịu tải

Dù đã băm nhỏ video thành HLS, 1000 user cùng request về một Origin Server thì server vẫn quá tải I/O ổ cứng. Đây là lúc CDN xuất hiện.

User 1 request
→
CDN Edge (Hà Nội)
→
Cache MISS → lấy từ Origin
→
Lưu cache tại Edge
999 Users còn lại
→
CDN Edge cache HIT
→
Origin không bị đụng đến

Khi User 1 xem video, CDN chưa có cache → lấy từ Origin (Cache Miss) và lưu các đoạn .ts tại Edge server gần nhất (Hà Nội, HCM…). Với 999 user còn lại, CDN trả thẳng từ Edge — nhanh, rẻ, Origin không bị đụng đến. Backend thực chất chỉ phục vụ đúng một người đầu tiên.

✅ Kết quả: Dù có 1 triệu user cùng xem, Origin Server của bạn cũng không hay biết. CDN như Cloudflare, AWS CloudFront hay Akamai sẽ gánh toàn bộ băng thông.

Bổ sung Cache Invalidation — Xoá cache khi cần thiết

CDN cache rất mạnh, nhưng cũng tạo ra một bài toán mới: nếu video bị xoá hoặc cần gỡ xuống khẩn cấp thì sao? CDN Edge vẫn đang giữ bản sao và tiếp tục phát cho user.

⚠️ Tình huống thực tế: Video vi phạm bản quyền được report. BE xoá khỏi S3 và DB ngay lập tức, nhưng CDN Edge tại hàng chục thành phố trên thế giới vẫn đang cache và phát video đó cho user trong nhiều giờ tiếp theo — gây rủi ro pháp lý nghiêm trọng.
Giải pháp — CDN Invalidation API: CloudFront có CreateInvalidation API, Cloudflare có Purge Cache API — gọi vào là cache bị xoá khỏi tất cả Edge nodes trong vòng 1–2 phút. Lưu ý một số CDN tính phí cho mỗi lần invalidate, vì vậy nên thiết kế URL có version trong path (ví dụ /v2/segment-001.ts) để chủ động kiểm soát cache thay vì phải invalidate thủ công.

Bổ sung Video Private — Signed URL & Signed Cookie

Không phải video nào cũng public. Google Drive, Dropbox, các hệ thống nội bộ đều có phân quyền xem. Lúc này CDN không thể cache công khai — cần cơ chế xác thực ngay tại tầng CDN.

  1. 1 Signed URL từ CDN: Thay vì trả link CDN thẳng, BE tạo một CloudFront Signed URL hoặc Cloudflare Signed Token — link có chữ ký HMAC, TTL ngắn (ví dụ 1 giờ), có thể gắn với IP của user để chống share link.
  2. 2 Signed Cookie (cho HLS): HLS gửi hàng trăm request để lấy từng segment .ts — dùng Signed URL riêng cho từng file là bất khả thi. Giải pháp là Signed Cookie — đặt cookie 1 lần, áp dụng cho toàn bộ domain CDN trong suốt session đó.
  3. 3 CDN tự xác thực ở Edge: Mỗi request đến Edge, CDN kiểm tra chữ ký mà không cần gọi về Origin. Hết hạn hoặc sai chữ ký → trả về 403 Forbidden ngay tại Edge, cực kỳ nhanh và không tốn tài nguyên Origin.
